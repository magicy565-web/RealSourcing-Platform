# 产品诊断报告 V2：打通“双边市场”自运转的任督二脉

**作者：Manus AI**

**日期：2026年2月21日**

---

## 1. 核心商业逻辑再确认

在您清晰的阐述下，我们对核心商业模式的理解已校准：

- **平台定位**：一个连接**海外采购商（买家）**和**国内制造商（工厂）**的 B2B 撮合平台。
- **买家价值**：解决 Sourcing 痛点，提供一个比传统渠道（线下展会、阿里巴巴）更高效、更值得信赖的供应商发现和连接工具。
- **工厂价值**：解决获客痛点，提供一个比传统渠道成本更低、更精准的触达真实海外买家的渠道。
- **核心载体**：Webinar (线上展会) 和 1:1 选品会 (深度对接)。

基于此，产品的核心是**构建一个能让买卖双方“自发、高效、低摩擦”地完成线上展会和深度对接的自运转市场**。

## 2. 产品现状诊断：双边市场的“断头路”

经过对代码库的再次深度审查，我们发现当前产品在实现“自运转”上存在几个致命的“断头路”：

### 卡点一：工厂侧有“责”无“权”，无法自发创造供给

工厂用户（`role="factory"`）虽然可以在系统中存在，但他们几乎**没有任何主动创造内容和发起活动的能力**。

- **无法发起 Webinar**：我们有 `webinars.create` 的后端路由，但**前端没有任何地方调用它**。工厂用户登录后，看到的 `Webinars.tsx` 页面和买家一样，只能“浏览”和“注册”别人（由运营手动创建）的 Webinar，完全无法“发起”自己的线上展会。
- **无法管理产品/工厂信息**：`FactoryDashboard.tsx` 只是一个纯静态的数据展示页面，工厂无法在这里**编辑工厂介绍、上传新产品、管理产品分类**。这导致供给侧的内容完全依赖运营手动录入数据库。

### 卡点二：买家侧有“意”无“门”，无法自助完成连接

买家虽然能浏览工厂，但**无法顺畅地完成从“发现”到“连接”的关键一步**。

- **无法自助预约会议**：`FactoryDetail.tsx` 页面上有“Start 1:1 Meeting”按钮，但它指向一个**不存在的路由 `/meeting/new`**。我们虽然有 `meetings.create` 的后端路由，但前端没有一个有效的、带日历选择功能的页面来承载它。买家看到心仪的工厂，想约个会，却发现是个“断头路”。

### 总结：当前的“双边市场”是一个伪命题

- **供给侧（工厂）**：被动等待运营投喂内容，无法自营销、自运营。
- **需求侧（买家）**：发现供给后，无法自助、高效地建立连接。

整个平台的运转，完全依赖于一个“上帝之手”（运营团队）在数据库中手动创建 Webinar、手动安排 Meeting。这违背了平台模式的初衷，无法形成网络效应。

## 3. MVP 重新定义：打通双边市场的“任督二脉”

我们的 MVP 目标必须聚焦于打通上述两个核心卡点，让平台能够**在没有运营干预的情况下，完成一次最基础的“工厂发起活动 → 买家参与 → 双方建立连接”的闭环**。

**MVP 核心用户故事：**

1.  **工厂自助入驻与内容管理**：工厂用户注册后，可以在自己的仪表盘（FactoryDashboard）里**完善工厂资料、上传和管理核心产品**。
2.  **工厂自助发起 Webinar**：工厂可以在仪表盘里**创建一个未来的 Webinar 活动**，包括填写标题、描述、上传封面、设定开播时间。
3.  **买家发现并参与**：买家在 `Webinars.tsx` 页面能看到工厂自助创建的 Webinar，并可以点击“注册”或直接进入直播间。
4.  **买家自助预约会议**：买家在 `FactoryDetail.tsx` 页面，可以点击“预约会议”按钮，进入一个**带有日历选择功能的会议预约页面**，向工厂发起一个 1:1 选品会邀约。
5.  **工厂响应邀约**：工厂在自己的仪表盘里收到会议邀约，并可以“接受”或“拒绝”。

### MVP 功能边界 (MoSCoW)

| 功能 | 类别 (MoSCoW) | 理由 |
|---|---|---|
| **工厂仪表盘 V2 (内容管理)** | **Must Have** | 解决供给侧内容创造的卡点 |
| **工厂自助创建 Webinar 功能** | **Must Have** | 打通“线上展会”自运转的核心 |
| **会议自助预约页面 (`/meeting/new`)** | **Must Have** | 打通“深度对接”自运转的核心 |
| **工厂仪表盘 V2 (会议管理)** | **Must Have** | 让工厂能响应买家的连接意图 |
| 真实视频流 (Agora/WebRTC) | **Won't Have** | MVP 阶段，视频区域保留 UI 占位符即可 |
| AI Meeting Reel | **Won't Have** | 属于连接建立后的“效率优化”，非核心闭环 |
| 复杂的权限和订阅系统 | **Won't Have** | 先跑通核心流程，再考虑商业化 |

## 4. 下一步行动计划：双边市场 MVP 重构

**目标**：用 2-3 周时间，完成能让买卖双方自运转的核心功能开发。

### P0: 供给侧赋能 (立即开始)

1.  **开发“创建/编辑 Webinar”页面**
    - **任务**: 创建一个新页面，让工厂用户可以填写表单（标题、描述、时间等）来调用 `webinars.create` 路由。在 `Webinars.tsx` 页面为工厂用户添加入口按钮。

2.  **重构 `FactoryDashboard.tsx`**
    - **任务**: 将其从静态展示页改造为动态管理后台。至少包含“我的产品”和“我的 Webinar”两个模块，允许工厂增删改查自己的产品和 Webinar 活动。
    - **依赖**: 后端需要提供 `products.create/update/delete` 和 `webinars.update/delete` 等路由。

### P1: 需求侧连接

1.  **开发 `/meeting/new` 会议预约页面**
    - **任务**: 创建一个带日历组件的页面，让买家可以选择日期和时间段，向指定工厂 `factoryId` 发起会议邀约，调用 `meetings.create` 路由。
    - **依赖**: `FactoryDetail.tsx` 中的按钮需要正确链接到此页面。

通过以上重构，我们将拥有一个真正具备“双边市场”雏形的 MVP。虽然功能极简，但它第一次实现了**供给（工厂）可以自发创造内容，需求（买家）可以自发建立连接**，这是平台能否存活并实现网络效应的生死线。
